# Мультиклиент: несколько клубов на одном сервере

Один процесс обслуживает несколько клубов, у каждого — собственные
Telegram-бот и/или сообщество ВК. Данные изолированы по клубам.

## Панель оператора (проще, чем curl)

Вместо ручных запросов ниже — веб-форма: `/admin/platform/login`, вход по
`ADMIN_API_TOKEN`. Дашборд со списком клиентов, кнопка «Добавить клиента»
(создаёт клуб + подключает боты + hot-reload за одну отправку формы),
быстрое продление оплаты прямо из таблицы. См. app/admin/platform.py.

## Как подключить нового клиента через API (на вашем сервере)

1. Клиент создаёт своего бота у @BotFather и/или сообщество ВК,
   передаёт вам токены.
2. Создайте клуб:  POST /api/tenants  {"name": "Клуб клиента"}
   (заголовок x-admin-token).
3. Задайте токены клуба:
   PATCH /api/tenants/{id}/tokens  {"tg_token": "...", "vk_token": "..."}
4. Переведите ботов на webhook (панель делает это сама для новых клиентов):
   PUT /api/tenants/{id}/delivery/tg  {"mode": "webhook"}
   PUT /api/tenants/{id}/delivery/vk  {"mode": "callback"}
5. Привяжите тренера клиента:
   PATCH /api/tenants/{id}/chat  {"admin_tg_id": ..., "admin_vk_id": ...}
6. Боты клиента поднимутся автоматически
   (hot-reload; в ответе будет "reloaded": true). Если по какой-то
   причине не поднялись — перезапустите сервис (Railway → Restart).
7. Тренер клиента пишет своему боту /start — видит меню управления.

Перед шагом 4 задайте `PUBLIC_BASE_URL` и случайный
`WEBHOOK_MASTER_SECRET` длиной не меньше 32 символов. API сначала
регистрирует адрес у Telegram/VK и только после успешного ответа сохраняет
режим. Секрет каждого клиента выводится из master-secret и токена и не
хранится отдельной открытой строкой в базе.

## Как это работает

- Токены хранятся в базе зашифрованно (`*_token_enc`).
- Новые клиентские боты работают через Telegram webhook и VK Callback API:
  мессенджер обращается к серверу только при реальном событии.
- Старые режимы polling/Long Poll остаются как обратимый резервный вариант;
  режим хранится отдельно для каждого клиента.
- Входящий webhook сначала коммитится в `inbound_events`, а затем
  обрабатывается worker'ом. Повтор одного `update_id/event_id` не создаёт
  повторной записи; зависшее событие после рестарта возвращается в очередь.
- Событие, пришедшее клиентскому боту, автоматически относится к его
  клубу (по токену) — участники и тренеры разных клубов не пересекаются.
- Исходящие (напоминания, рассылки, публикации) уходят через бота
  того клуба, которому принадлежит адресат.
- Бот из переменных окружения (TG_TOKEN/VK_TOKEN) работает как раньше —
  это «клуб №1» владельца платформы.

## Оплата клубов (SaaS)

- PATCH /api/tenants/{id}/billing  {"paid_until": "2026-08-01"}
  — до какой даты клуб оплачен. Пустая строка — без ограничений.
  Через панель оператора это же делается кнопкой «Продлить» в дашборде.
- После даты боты и веб-страница клуба отвечают
  «работа приостановлена»; напоминания и автосоздание тренировок
  не выполняются.
- **Уведомления.** Раз в сутки (фоновая задача):
  - владелец платформы получает в Telegram сводку по всем клубам,
    у которых оплата истекает в ближайшие 3 дня или уже истекла
    (нужен PLATFORM_OWNER_TG_ID — см. .env.pro.example);
  - **клиент** (тренер/владелец клуба) получает уведомление в СВОЁМ
    боте — одно при входе в трёхдневное окно «скоро истекает» и одно
    при фактическом истечении. Не спамит каждый день: повторно придёт
    только после продления (нового paid_until) и приближения новой даты.
    Требует, чтобы у клуба был задан admin_tg_id и/или admin_vk_id.
  - текст уведомления клиенту можно дополнить контактом для продления —
    переменная PLATFORM_SUPPORT_CONTACT (например "@ваш_ник"); без неё
    используется общая формулировка «свяжитесь с администрацией сервиса».

## Ограничения

- Для автоматической настройки VK Callback API токен сообщества должен иметь
  права управления Callback API. Панель получает confirmation code,
  регистрирует сервер и включает события `message_new` и `message_event`.
- При замене токена webhook-бота сначала верните его в polling/Long Poll,
  замените токен и снова включите webhook. Это не даёт оставить внешний
  webhook со старым производным секретом.
- Для аварийного отката используйте те же endpoints с режимами `polling` и
  `longpoll`; pending updates при переключении не удаляются.
