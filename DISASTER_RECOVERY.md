# План восстановления: что делать, если Railway недоступен

Короткая версия: у тебя есть свежий бэкап базы в Telegram (себе), код лежит
в GitHub — переезд на другой хостинг занимает 15–30 минут без потери
данных. Ниже — по шагам.

---

## 0. Что уже подстраховано заранее

- **Бэкап базы раз в сутки** приходит тебе в Telegram файлом (от
  платформенного бота) — работает независимо от Railway, переживает его
  полное падение. Настраивается переменной `PLATFORM_OWNER_TG_ID` в
  Railway (твой Telegram ID).
- **Бэкап по требованию** — кнопка «💾 Создать бэкап сейчас» в
  `/admin/platform` (панель оператора), если нужен самый свежий снимок
  прямо сейчас, не дожидаясь ночного прогона.
- **Код** — в GitHub (`github.com/Hydwild/sports-training-bot`), не
  зависит от Railway вообще.
- **Боты работают через polling**, не webhook (`TG_MODE=polling`,
  `RUN_VK_POLLING=true`) — это значит, что после переезда боты сами
  переподключатся к Telegram/VK с новой машины, никакой DNS или
  webhook-адрес перерегистрировать не нужно. Единственное, что зависит от
  адреса — публичная страница записи (`/club/{id}`) и вход в веб-админку
  через Telegram Login Widget (см. шаг 4).

---

## 0.2. Проверка, что копия рабочая

Файл бэкапа, который «похож на дамп», и файл, из которого реально можно
восстановиться, — не одно и то же. Поэтому не полагайся на то, что копия
приходит: проверяй её восстановлением.

- **В панели оператора** (`/admin/platform`) — кнопка «🧪 Проверить
  восстановление». Она снимает свежую копию и восстанавливает её во
  ВРЕМЕННУЮ базу (рабочую не трогает), затем показывает дату последней
  успешной проверки. Делай это хотя бы раз в месяц.
- **Вручную из скачанного файла** — распакуй (`scripts/backup_tool.py
  --decrypt`) и восстанови в тестовую базу, как в шаге 2.3, но с ДРУГИМ
  именем базы. Никогда не восстанавливай поверх рабочей вслепую.

Проверка на PostgreSQL требует `psql` в окружении и служебного доступа для
`CREATE DATABASE`/`DROP DATABASE`; тестовая база создаётся со случайным
именем `drill_...` и удаляется только она.

## 0.1. Ключи, без которых бэкап бесполезен

Копия базы содержит **зашифрованные** телефоны клиентов. Ключ в неё не
попадает — иначе шифрование не имело бы смысла. Значит, восстановление
возможно только если у тебя отдельно сохранены:

| Переменная | Что откроет | Где хранить |
|---|---|---|
| `PHONE_ENC_KEY` | телефоны веб-клиентов (версия `v1`) | менеджер паролей, НЕ в Railway-only и не в чате с бэкапами |
| `PHONE_KEYRING` | телефоны, зашифрованные прежним ключом (`jwt:<старый JWT_SECRET>`) | там же; нужен только пока не завершён переход |
| `JWT_SECRET` | телефоны версии `jwt`, если связка не заполнена | там же |
| `BOT_TOKEN_ENC_KEY` | токены Telegram/VK клубов | там же |
| `BOT_TOKEN_KEYRING` | токены, зашифрованные прежним ключом | там же |
| `BACKUP_ENC_KEY` | **саму резервную копию** | там же — но НЕ в том чате, куда приходят копии |

Потерян ключ — записи и имена восстановятся, телефоны нет. Приложение при
этом поднимется: карточка участника откроется без номера, а не упадёт.

### Порядок перехода на выделенный ключ телефонов

Нельзя выполнять одним деплоем — иначе клиенты «пропадут» из поиска и
заведутся заново дублями.

1. Выкатить код, который читает обе версии (уже в `main`).
2. Задать `PHONE_ENC_KEY` **и** `PHONE_KEYRING=jwt:<текущее JWT_SECRET>`.
3. `python -m scripts.migrate_phone_keys --dry-run` — посмотреть план.
   Команда не печатает телефоны и ключи, только количества и id.
   Код возврата `2` означает «есть нечитаемые строки или клиенты-дубли»:
   применять нельзя, пока не разберёшься.
4. Снять бэкап.
5. `python -m scripts.migrate_phone_keys --apply` — идемпотентно.
6. `python -m scripts.migrate_phone_keys --verify` — должно быть
   «отстали 0, не расшифрованы 0».
7. Только теперь можно убрать `PHONE_KEYRING` и ротировать `JWT_SECRET`.

### Порядок перехода на шифрование токенов ботов

1. Задать `BOT_TOKEN_ENC_KEY`.
2. `python -m scripts.migrate_bot_tokens --dry-run`.
3. Снять бэкап.
4. `python -m scripts.migrate_bot_tokens --apply` — шифрует и очищает
   открытые колонки.
5. `python -m scripts.migrate_bot_tokens --verify` — «осталось открытым
   текстом 0, не расшифровано 0».
6. **Перевыпустить токены** у @BotFather (`/revoke`) и в настройках
   сообщества VK: прежние значения успели побывать в дампах, а дампы
   уходили в Telegram. Новые сохранить через панель оператора — они сразу
   лягут зашифрованными.
7. Удаление самих колонок `tenants.tg_token` / `vk_token` — отдельной
   миграцией позже, когда убедитесь, что боты поднимаются.

## 1. Найди последний бэкап

Открой чат с платформенным ботом в Telegram (личка, не тестовые клубы) —
последнее сообщение с файлом `backup_ГГГГ-ММ-ДД.sql.gz.enc` (Postgres) или
`backup_ГГГГ-ММ-ДД.db.gz.enc` (SQLite). Скачай на компьютер.

Файл зашифрован. Сначала проверь, что он читается твоим ключом, потом
расшифруй:

```bash
python -m scripts.backup_tool --verify backup_2026-07-22.sql.gz.enc
python -m scripts.backup_tool --decrypt backup_2026-07-22.sql.gz.enc --out dump.sql.gz
```

`--verify` печатает SHA-256 (сверь с подписью под файлом в Telegram) и
говорит, что внутри. Инструмент никогда не пишет в рабочую базу —
восстановление ниже выполняется отдельно и осознанно.

Если бота тоже не поднять (Railway полностью недоступен, включая логи) —
ищи файл во входящих Telegram на любом устройстве, где стоит клиент,
сообщения там не теряются.

## 2. Новый хостинг: Render (рекомендованный вариант)

Почему Render: тот же принцип, что и Railway (Docker-образ из GitHub +
managed Postgres), минимум переделок.

### 2.1. Создать Postgres

1. Render Dashboard → **New** → **PostgreSQL**
2. Любое имя, ближайший регион, план подходящий по нагрузке (Free/Starter
   для начала — апгрейднуть можно позже без даунтайма)
3. После создания скопируй **Internal Database URL** (или External, если
   восстанавливаешь с локального компьютера — см. шаг 2.3)

### 2.2. Создать Web Service

1. Render Dashboard → **New** → **Web Service**
2. Подключи GitHub-репозиторий `sports-training-bot`
3. Render сам увидит `Dockerfile` в корне — оставь **Environment: Docker**
4. Instance type — Starter достаточно для старта

### 2.3. Восстановить базу из бэкапа

Из терминала (нужен установленный `psql`/`pg_dump` локально, или можно
выполнить прямо из Render Shell после первого деплоя):

```bash
gunzip -c backup_2026-01-01.sql.gz | psql "postgresql://user:pass@host:port/dbname"
```

Строку подключения (`postgresql://...`) возьми из Render → твоя база →
**External Database URL** (для восстановления с локальной машины) — после
восстановления Web Service будет ходить по Internal URL, он быстрее.

Если бэкап был `.db.gz` (SQLite, с Lite-инстанса) — это SQLite-файл, в
Postgres он так не восстановится; такой бэкап заливается на volume Render
(Disk) как обычный файл, схема БД пересоздастся сама при старте
приложения (`create_all` + `_ensure_columns` в `app/main.py`).

### 2.4. Переменные окружения

Скопируй из Railway → Variables ВСЁ, что было (список см.
`.env.pro.example`), и добавь/поменяй:

```
DATABASE_URL=<Internal Database URL от Render, с postgresql+asyncpg://>
PUBLIC_BASE_URL=https://<имя-сервиса>.onrender.com
```

⚠️ `DATABASE_URL` от Render идёт в формате `postgresql://...` —
допиши `+asyncpg` после `postgresql`, иначе приложение не подключится:
`postgresql+asyncpg://user:pass@host:port/dbname`.

Остальные переменные (`JWT_SECRET`, `ADMIN_API_TOKEN`, `TG_TOKEN`,
`VK_TOKEN`, `PLATFORM_OWNER_TG_ID` и т.д.) переносятся как есть — это
важно: **не генерируй новый `JWT_SECRET`**, иначе все текущие сессии
клиентов (веб-админка) слетят и понадобится заново логиниться (не
критично, но лишнее неудобство). Токены ботов (`TG_TOKEN`/`VK_TOKEN`)
тоже переносятся как есть — тот же бот, тот же токен, просто теперь
опрашивает VK/Telegram с другой машины.

### 2.5. Первый деплой и проверка

1. Render соберёт образ и запустит `start.sh` (тот же, что на Railway) —
   он сам применит миграции Alembic (`EDITION=pro`)
2. Проверь `https://<имя-сервиса>.onrender.com/health` → должно быть
   `{"status":"ok",...}`
3. Напиши боту `/start` в Telegram — должен ответить (polling сам
   переподключился, ничего дополнительно включать не нужно)
4. Открой `/admin/platform/login`, проверь список клиентов — база должна
   быть восстановлена полностью (те же клубы, тренировки, участники)

### 2.6. Что нужно поменять у клиентов вручную

Единственное, что реально зависит от домена — публичная страница записи
и QR-коды, которые ты уже отдал клиентам (там зашит старый URL). Если
адрес сменился (`onrender.com` вместо `up.railway.app`):
- новый QR-код на каждую страницу — `/club/{id}/qr` уже отдаёт актуальный
  URL сам (использует `PUBLIC_BASE_URL`), просто перепечатать/переслать
- старые распечатанные QR в залах клиентов продолжат вести на мёртвый
  Railway-адрес — предупреди клиентов, что нужно заменить

## 3. Альтернатива: вернуться на Railway

Если проблема была временной (сбой конкретного сервиса/региона, а не
полное исчезновение аккаунта) — Railway обычно восстанавливается сам.
Данные на Volume (SQLite) или managed Postgres переживают рестарт
сервиса — трогать ничего не нужно, просто дождаться. Полный переезд (этот
документ) нужен только если аккаунт/сервис Railway пропал безвозвратно.

## 4. Чек-лист после любого восстановления

- [ ] `/health` отвечает `200 OK`
- [ ] Боты отвечают на `/start` (TG и VK, если подключён)
- [ ] `/admin/platform` показывает всех клиентов с прежними данными
- [ ] Публичная страница `/club/{id}` открывается, тренировки на месте
- [ ] `PLATFORM_OWNER_TG_ID`/`PLATFORM_SUPPORT_CONTACT` перенесены —
      бэкапы и биллинг-уведомления продолжат работать
- [ ] Первый ручной бэкап после переезда («💾 Создать бэкап сейчас») —
      убедиться, что цепочка бэкапов не прервалась
